--------------------------------------------------------------------------------

El plan actual no esta listo para implementacion perfecta sin ajustes.

La direccion general es correcta:
- migrar fuera del monolito ASP.NET MVC
- usar backend moderno
- usar SQL Server
- proteger escrituras con JWT
- controlar stock con transacciones
- guardar fotos en disco, no en base de datos
- tener web administrativa y app movil para campo

Pero faltan piezas criticas:
- contrato SSO definitivo para entrar desde Portal
- diccionario real de Excel y reglas de limpieza
- carga normalizada de catalogo con imagenes
- modelo SQL final con constraints e indices completos
- decision tecnica sobre TypeORM vs `mssql` directo
- contrato API detallado por endpoint
- plan de implementacion por fases verificables
- plan de pruebas de importacion y concurrencia
- despliegue ajustado al VPS Linux/Nginx/PM2 real

2. ESTADO REAL DEL PROYECTO
--------------------------------------------------------------------------------

Ruta analizada:
- `/opt/apps/asistencia/registro`

El repo contiene cuatro grupos:

1. Legado ASP.NET MVC:
   - `Controllers/`
   - `Views/`
   - `Models/`
   - flujo real de evento y despacho en `EventRegistrationController.cs`

2. Proyecto Nest:
   - `/nest`
   - esta practicamente en estado inicial generado por CLI
   - no tiene modulos reales de auth, attendance, catalog, dispatch ni SQL Server

3. Proyecto React:
   - `/react`
   - esta en plantilla inicial Vite
   - no implementa dashboard, login SSO ni consumo de API

4. Proyecto Flutter:
   - `/flutter`
   - esta en plantilla inicial Flutter counter
   - no implementa scanner, cache, despacho ni login

Diagnostico:
La documentacion describe un sistema objetivo, pero el codigo nuevo todavia no
representa ese sistema. La implementacion debe comenzar realmente desde cero
en `nest`, `react` y `flutter`, usando el legado solo como referencia funcional.

--------------------------------------------------------------------------------
3. HALLAZGOS DEL SISTEMA LEGADO
--------------------------------------------------------------------------------

Archivo principal:
- `Controllers/EventRegistrationController.cs`

Vista principal:
- `Views/EventRegistration/DespachoDiaNino.cshtml`

Flujo legado real:

1. `DespachoDiaNino()`
   - renderiza la pantalla de despacho.

2. `GetHijosParaDespacho(carnet)`
   - consulta `dbo.tblDespachoJuguetesDiaNino`
   - cruza contra `dbo.tblCatalogoJuguetesDiaNino`
   - asigna juguete por `Categoria` y `Genero`
   - usa fallback `TODOS` si no hay juguete especifico por genero

3. `RegistrarEntregaJuguete(id, entregado)`
   - actualiza `Entregado`, `FechaEntrega`, `UsuarioDespacho`
   - no valida stock
   - no usa transaccion de inventario
   - no registra receptor fisico
   - no guarda motivo de reversion

4. `ObtenerResumenInventario()`
   - calcula entregados/pendientes desde la tabla de censo
   - no descuenta `StockActual`

5. `ObtenerCatalogoCompleto()`
   - devuelve catalogo con `FotoBase64`

6. `GuardarJugueteCatalogo()`
   - inserta o actualiza juguetes
   - valida duplicado por categoria/genero
   - almacena foto en base de datos como Base64

7. `EliminarJugueteCatalogo()`
   - hace delete fisico

Riesgos confirmados:
- el stock no esta protegido
- hay imagenes Base64 en SQL Server
- no hay auditoria completa de receptor
- el controlador mezcla demasiadas responsabilidades
- el flujo no esta preparado para alta concurrencia

6. DECISION TECNICA RECOMENDADA
--------------------------------------------------------------------------------

Backend:
- NestJS, pero con acceso SQL Server mediante `mssql` directo.
- Evitar TypeORM en la primera version critica.
- Usar procedimientos o queries parametrizados.
- Para despacho/reversion usar transacciones explicitas.

Frontend web:
- React + Vite.
- Dashboard administrativo primero.
- No construir marketing ni pantalla decorativa.
- Enfocarse en busqueda, inventario, importacion, reportes y auditoria.

Movil:
- Flutter.
- Primera version online estable.
- Offline cache queda para segunda fase si el evento lo exige.
- El scanner y el flujo de entrega deben ser mas importantes que animaciones.

SSO:
- Reutilizar el patron documentado en `solicitud`.
- Token del Portal firmado con `SSO_SECRET`.
- Backend emite JWT interno de `asistencia`.
- Validar `type` o `token_type = SSO_PORTAL` para evitar aceptar JWT genericos.

Imagenes:
- Extraer imagenes del Excel de juguetes a archivos.
- Convertir a WebP.
- Guardar ruta en SQL Server.
- No guardar Base64 en SQL Server.
